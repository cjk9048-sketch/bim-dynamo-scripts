# 자문 요청 — 3D 폴리선(닫힌 루프) 2개의 정확한 합집합

## 배경 (Civil 3D 2026 부지정지 플러그인, C# + NetTopologySuite 2.x)

계단식 절토/성토 정지 설계에서 **정지경계(finalRing)** 를 만든다:

- **원지반 TIN**(측량, ~68,000 삼각형)과 **가상 절토/성토 TIN**(설계, ~15,000 삼각형)의
  **교차선(daylight)** 을 삼각형쌍 높이차 d=0 보간으로 추출 → 폴리라인 체인들.
- 체인의 열린 끝을 최근접끼리 이어 **닫힌 3D 루프**로 만든다. ← **여기까지는 검증 완료(정확함)**.
  - 예: 절토 = 닫힌 루프 1개(~2,960㎡), 성토 = 닫힌 루프 3개(1,111 + 143㎡ 등).
- 각 루프의 정점은 3D(X, Y, Z=지반고). 계획폴리곤(pad 외곽)은 8점 닫힌 3D 폴리선(Z=설계고, 구간별 상이).

## 하려는 것 — 합집합이 필요한 이유

`최종경계 = 닫힌교선 ∪ 계획폴리곤` 이 필요하다:
1. 절/성토가 **2개 이상 조각**일 때 계획(pad)으로 이어 **한 경계로 병합** (Civil TIN Outer 경계는 1개).
2. 계획과 안 닿는 **잡루프 제외**.

## 문제 (핵심 버그) — “계획직선 sticking”

계획폴리곤 **전체**와 합집합하면, 절토-성토 **전이선**(교선이 계획 내부를 대각선으로 지나는 구간)이
합집합 외곽선에서 사라지고 **계획폴리곤의 직선 변으로 대체**된다.
(합집합의 외곽선 = own ∪ plan 이므로, plan이 반대편(성토) 구역까지 덮어 전이선이 내부로 삼켜짐 — 기하적으로 당연한 결과)

증상: 초록 경계가 지형(등고선)을 따라 내려오다 전이 지점에서 갑자기 계획 직선에 “붙어”버림.

## 실패한 시도들

| 시도 | 방법 | 결과 |
|---|---|---|
| 1 | own ∪ plan 전체 (기존) | 전이선이 계획직선으로 덮임(sticking) |
| 2 | (plan − 반대면 면조각).Union | 면조각(d≈0 접선지대에서 불완전) 탓 경계 파편화 |
| 3 | 채우고-깎기: (own∪plan).Difference(반대 면조각) | 면적은 정답과 일치했으나 형상은 여전히 sticking(면조각 불완전) |
| 4 | 교선 체인으로 plan을 polygonize 분할 | 전이선이 plan 내부에 떠 있어(양끝이 plan 경계에 안 닿음) 분할 실패/오분할 |
| 5 | 격자 diff=0 마칭스퀘어 경계 | 모양은 정확하나 0.25~0.5m 격자 잔차로 표면과 미세 불일치(뭉둥그려짐) |

교훈: **면조각(area) 기반은 전이선(접선지대)에서 신뢰 불가. 정확한 것은 오직 “교선 체인(선)” 뿐.**

## 현재 구현 (검토 요청 대상)

```
절토 경계 = 절토교선루프들 ∪ ( 계획폴리곤 − Union(성토교선루프들) )
성토 경계 = 성토교선루프들 ∪ ( 계획폴리곤 − Union(절토교선루프들) )
```

- 반대편을 뺀 계획부분(planPart)과 자기 루프를 UnaryUnion → 외곽선만 채택(plan과 intersects 안 하면 제외).
- **Z 복원(자문 반영)**: 합집합은 2D(XY)로 수행하고, 결과 정점의 Z는 **원본 3D 폴리선(자기교선·반대교선·계획선) 중
  최근접 선분에서 선형보간**으로 역투영.

```csharp
public static List<List<Point3>> UnionLoopsWithPlan(
    IReadOnlyList<List<Point3>> ownLoops, IReadOnlyList<List<Point3>> oppLoops,
    IReadOnlyList<Point3> plan, CachedGroundSurface ground, out string diag)
{
    // 닫힌 루프 → NTS 폴리곤(무효면 Buffer(0))
    Geometry? ToPolyG(List<Point3> lp) { /* LinearRing 생성, IsValid 아니면 Buffer(0) */ }

    var ownGs = ownLoops.Select(ToPolyG).Where(g => g != null).ToList();
    Geometry planG = CreatePolygon(plan).Buffer(0);

    Geometry planPart = planG;
    var oppGs = oppLoops.Select(ToPolyG).Where(g => g != null).ToList();
    if (oppGs.Count > 0)
        planPart = planG.Difference(UnaryUnionOp.Union(oppGs)); // 계획 − 반대편

    var u = UnaryUnionOp.Union(ownGs.Append(planPart));         // 자기 ∪ 계획나머지

    // 외곽선 채택: Polygon && Area>0.5 && Intersects(planG). Z는 원본 폴리선 최근접 보간(역투영).
}
```

## 우려 지점 / 질문

1. **전이선 좌표 일치 문제**: `planG.Difference(oppUnion)`의 전이선 변은 성토 루프 좌표에서 온다.
   own(절토) 루프의 전이 구간 좌표와 이론상 같은 선(design=ground)이지만 **서로 다른 TIN에서 독립 계산**되어
   수 mm~cm 어긋날 수 있다. 그 상태로 `Union(own, planPart)` 하면 그 틈이 **실오라기 슬리버/자기접촉 링**으로
   남을 수 있는데, 이를 견고하게 봉합하는 표준 기법은? (SnapRoundingNoder? Snap-to tolerance된 OverlayNG?
   `GeometryPrecisionReducer`? 두 선을 먼저 스냅 병합?)
2. NTS에서 `OverlayNGRobust` / `UnaryUnionOp` 사용 시 **고정 정밀도 모델(PrecisionModel, 예: 1mm 격자)** 을 쓰면
   위 미세 어긋남이 자동 봉합되는가? 부작용(정점 이동으로 인한 자기교차)은?
3. 결과 외곽선이 Civil 3D `AddBoundaries(Outer)` 가 받아들이는 조건(자체교차 없음·닫힘)을 항상 만족하게 하는
   후처리 권장 순서는? (현재: 무효 시 Buffer(0), 최후 5mm 침식-팽창)
4. 위 접근 자체보다 더 표준적인 대안이 있으면 제안 바람. (예: 두 교선 체인 + 계획선을 노드로 잇는
   선형 network → Polygonizer, 이때 전이선 양끝이 계획선에 닿도록 스냅 연장하는 방식 등)

## 참고 수치 (실측)

- 계획폴리곤: 8점, 3,950㎡, Z 100(구간별 상이 가능)
- 절토 닫힌 교선: 1루프 812점, ~2,960㎡ / 성토: 3루프(총 ~1,254㎡, 최대 1,111㎡)
- 전이선: 계획 내부를 지나는 대각선, 양끝이 daylight와 이어짐(계획 경계에는 안 닿음)
- 과거 교선 원시 체인 사이 틈: 최대 ~15m(접선지대, 틈메움 로직으로 연결)
